# L-2: Token 만료 사전 체크 — 미구현 결정

> 결정일: 2026-03-22

## 배경

코드 리뷰(2026-03-20)에서 JWT 토큰 만료 전 사전 갱신(proactive refresh) 미구현 발견.

## 현재 방식 (Reactive)

```
API 요청 → 서버 401 → ApiClient 인터셉터가 자동 refresh → 원래 요청 재시도
```

`ApiClient`에 이미 401 감지 + 자동 갱신 + 재시도 로직이 구현되어 있어 사용자는 실패를 인지하지 못함.

## Proactive 방식 (미채택)

```
API 요청 전 → JWT exp 확인 → 만료 1-2분 전이면 미리 갱신 → 항상 유효한 토큰으로 요청
```

## 미구현 결정 이유

1. **UX 차이 체감 불가**: 토큰 만료는 보통 1시간에 1번. Reactive 방식의 추가 지연은 ~200ms (401 왕복 1회). 하루 몇 번 200ms 차이를 사용자가 인지하기 어려움.

2. **현재 구조가 이미 안정적**: `ApiClient`의 401 인터셉터 + `Completer` 기반 중복 갱신 방지가 잘 작동 중. 추가 복잡도 대비 효과 낮음.

3. **구현 비용**: JWT payload 파싱 + 만료 시간 계산 + 타이머 관리 + 서버-클라이언트 시간 차이 보정 필요. 현재 방식 대비 코드 복잡도만 증가.

## 결론

현재 reactive 방식이 충분히 잘 작동하므로 사전 갱신은 불필요. 향후 토큰 만료 주기가 극도로 짧아지거나 (예: 5분), 실시간 연결(WebSocket)이 추가되면 재검토.
