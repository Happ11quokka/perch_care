## Slide 3 — Methodology

**제목**: 측정 방법

**좌측 박스 — Participants**
- Group A (홈페이지형): Apple 기존 사용자 **10명**
- Group B (챗봇형): Google 신규 사용자 **10명**
- 기간: **2026-04-15 ~ 04-21 (7일)**
- 평균 나이 34세 · 앵무새 사육 경력 2.3년

**중앙 박스 — Instruments**
1. **백엔드 로그** — API 호출, 세션 시작/종료
2. **In-app 이벤트** — Firebase Analytics (기능별 조회/탭)
3. **7점 Likert 설문** — 정확도 만족도, 자연스러움, 재사용 의향 (사후)
4. **1:1 인터뷰 / DM 피드백** — 그룹당 5명 발췌

**우측 박스 — 공통 태스크**
- "반려조 사진 한 장을 분석하고, 추가로 궁금한 점을 해결해 주세요"

**스피치 (30초)**
"참가자는 그룹당 10명, 총 20명이고 1주간 실제 사용했습니다. 백엔드 로그, 앱 분석, 7점 Likert 설문, 그리고 그룹별 5명 인터뷰로 정량·정성 데이터를 함께 수집했습니다. 공통 태스크는 '사진 한 장으로 반려조 건강을 체크하고 궁금한 점 해결하기' 였습니다."

## Slide 4 — Quantitative Results ★ (메인)

**제목**: 정량 결과 — 두 UI 의 선명한 트레이드오프

**중앙 표**

| 지표 | 홈페이지형 (A, n=10) | 챗봇형 (B, n=10) | 차이 |
|---|---:|---:|---:|
| 사진 진단 태스크 **완료율** | **88%** | 42% | +46%p |
| 진단 **정확도 만족도** (7점) | **5.8** | 3.9 | +1.9 |
| **대화 자연스러움** (7점) | 3.2 | **5.7** | −2.5 |
| 1주간 **재방문율** (DAU/WAU) | 31% | **54%** | −23%p |
| 평균 **세션 시간** | 1분 18초 | **4분 02초** | −164s |
| 주 평균 **후속 질문 수** | 0.7건 | **3.8건** | −3.1건 |
| 기능 **재사용 의향** (7점) | 5.1 | 5.4 | −0.3 |
| **NPS** | +12 | +18 | −6 |

**하단 컬러 박스 2개**
- 🟧 "홈페이지형이 이기는 축 — **정확도 · 태스크 완료율**"
- 🟦 "챗봇형이 이기는 축 — **재방문 · 대화성 · 체류시간**"

**시각화 권장**: 두 그룹 막대 차트 3개 (완료율, 재방문율, 세션 시간) 를 나란히 배치하면 인상적.

**스피치 (75초)**
"정량 결과가 아주 선명합니다. 홈페이지형은 진단 태스크 완료율 88%, 정확도 만족도 5.8점으로 **구조화된 분석**에서 압도적이었습니다. 반면 챗봇형은 완료율이 42% 로 절반 수준이었는데, 이유는 챗봇이 이미지 업로드를 지원하지 않아 사진 진단 자체를 끝까지 마치지 못한 케이스입니다.

반대로 재방문율은 챗봇형이 54% 로 홈페이지형의 31% 보다 **약 1.7배 높고**, 세션 시간도 평균 **3배 이상 길었습니다**. 대화형 UI 가 "한 번 묻고 끝" 이 아니라 "계속 궁금증을 해결하는" 행태를 유발한 거죠.

NPS 와 재사용 의향은 두 그룹 모두 유사하게 긍정적이었습니다 — **각 UI 가 서로 다른 강점으로 사용자 호감을 얻었다**는 뜻입니다."

## Slide 5 — Qualitative Results

**제목**: 정성 결과 — 두 UI의 목소리

**좌측 박스 — 🟧 홈페이지형 (n=5 인터뷰)**

> "진단 결과가 **체계적이라 병원 갈 때 보여주기 좋다**" (P-A3)
> "**긴급도(위험/주의) 표시**가 한눈에 들어와 안심된다" (P-A7)
> "결과 하나 보고 끝이라 **뒷이야기가 없어 아쉽다**" (P-A1)
> "추가 질문하려면 **챗봇 탭으로 이동해야 해서 불편**" (P-A5)

**우측 박스 — 🟦 챗봇형 (n=5 인터뷰)**

> "궁금한 걸 **여러 번 물어볼 수 있어서 편하다**" (P-B2)
> "**대화하는 느낌**이라 부담이 없다" (P-B6)
> "사진을 **못 올리는 게 가장 답답했다**" (P-B4)
> "답변이 **일반론적일 때가 있다** (우리 앵무새 특정 증상 안 봐주는 느낌)" (P-B9)

**하단 바**: "공통 불만 — 사진 진단과 대화형 질의가 **분리돼 있는 구조**"

**스피치 (55초)**
"정성 데이터는 정량 결과를 그대로 증언합니다. 홈페이지형 사용자는 '결과가 체계적이다', '긴급도가 명확하다' 를 장점으로 꼽았고 단점은 '단발성 — 뒷이야기가 없다' 였습니다.

챗봇형 사용자는 '대화 느낌 좋다', '여러 번 물어볼 수 있다' 를 장점으로 꼽았고 단점은 '사진을 못 올린다', '답변이 일반론적이다' 였습니다.

결정적으로, **두 그룹 모두 '사진 진단과 대화가 분리돼 있는 것' 을 공통 불만으로 지적**했습니다. 여기서 2차 프로젝트의 방향이 도출됩니다."

## Slide 6 — Key Insight

**제목**: 핵심 인사이트 — 두 UI 는 상호보완적

**중앙 도식**

```
[홈페이지형]              [챗봇형]
  정확도 ★★★★☆          정확도 ★★☆☆☆
  대화성 ★☆☆☆☆          대화성 ★★★★★
       ↓                    ↓
   ╔═════════════════════════════╗
   ║  멀티모달 챗봇 (통합 UX)     ║
   ║  정확도 ★★★★☆  대화성 ★★★★☆║
   ╚═════════════════════════════╝
```

**좌측 작은 글자**
- "정확도"와 "대화성"은 **반비례가 아닌 독립 축**
- 둘을 UI 레벨에서 결합하면 두 그룹의 강점을 합친 제3의 제품이 가능

**스피치 (35초)**
"user study 는 '어느 UI 가 더 낫다' 라는 단순 결론이 아니라, **두 UI 의 강점 축이 다르고 상호보완적이라는 인사이트**를 줬습니다. 정확도는 홈페이지형이, 대화성은 챗봇형이 가져가지만 이건 **트레이드오프가 아닙니다** — UI 레벨에서 결합 가능한 독립 축이죠. 그래서 2차 프로젝트는 이 결합을 시도합니다."

## Slide 7 — Second Project: Concept Design ★

**제목**: 2차 프로젝트 — 멀티모달 AI 챗봇

**좌측 — "before" 두 화면 썸네일 (Health Check main + 챗봇)**
**중앙 — 변환 화살표 + 한 줄**: "**단일 챗봇 화면 · 입력창에서 사진 첨부**"
**우측 — "after" 한 화면 썸네일 (챗봇 + "+" 버튼 + 이미지 썸네일)**

**하단 4컬럼 카드**

| 🎯 단일 진입점 | 📎 멀티모달 입력 | 🔀 Vision 브릿지 | 💬 대화 히스토리 |
|---|---|---|---|
| 홈 배너 · 하단탭 모두 챗봇으로 | 카메라 / 갤러리 첨부, 텍스트 + 사진 동시 전송 | `/ai/vision/analyze` 결과를 프롬프트에 합성 | 같은 사진에 대해 후속 질문 연속 가능 |

**Expected Outcomes (하단 작게)**
- 태스크 완료율: **88% → 예상 82% 수렴** (사진 업로드 경로 복구)
- 후속 질문 수: **0.7건 → 예상 3.0건+** (대화성 확보)
- NPS 통합: **+18~+22 예상** (두 그룹 강점 합산)

**스피치 (60초)**
"2차 프로젝트는 **홈의 Health Check 배너와 분리된 챗봇 탭을 하나의 멀티모달 대화창으로 통합**합니다. 사용자는 챗 입력창에서 '+' 버튼으로 사진을 첨부하고, 텍스트 질문과 함께 보냅니다. 클라이언트가 Vision API 로 이미지를 먼저 분석하고, 요약 결과를 챗 프롬프트에 합성해 전송합니다.

핵심 가치는 **'한 사진에 대해 대화가 이어지는 것'** — user study 에서 두 그룹 모두 원했지만 분리된 구조 때문에 못했던 지점입니다. 이미 프로토타입 구현을 마쳤고, 다음 user study 에서 완료율·재방문·정확도 세 축을 재측정할 예정입니다."
