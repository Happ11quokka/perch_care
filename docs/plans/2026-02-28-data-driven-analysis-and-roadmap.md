## 2. 핵심 발견사항

#### A. 플랫폼 접근성 문제 — **17건 (가장 많은 요청)**

| 유형 | 건수 | 상세 |

### 2-2. DB 사용률 기반 기능 분석

```
기능별 사용률 (총 가입자 250명 대비)

체중 기록    ████████████████████████ 98.4% (246건)
사료 기록    ████                     16.8% (42건)
음수 기록    ███                      10.4% (26건)
AI 채팅     ██                        9.6% (24건)
일일 기록    █                         5.6% (14건)
스케줄      █                         3.6% (9건)
AI 건강체크  ▏                         0.0% (0건)
```

**핵심 인사이트:**

1. **체중 기록이 킬러 기능** — 거의 모든 사용자가 사용. 이 기능의 완성도가 리텐션 핵심.
2. **사료/음수 사용률 극히 낮음** — 입력 UX가 복잡하거나 동기부여 부족. 간소화 또는 gamification 필요.
3. **AI 기능 사용률 10% 미만** — 발견 가능성(discoverability) 부족이거나, 기능 자체의 매력도 부족. 홈 화면 노출 강화 필요.
4. **일일 기록/스케줄 거의 미사용** — 온보딩 없이 기능만 존재. 사용 동기 부여 전략 필요.
5. **푸시 알림 등록 2건** — 알림 권한 요청 타이밍/UX 문제. 1,991건 알림 발송 중이지만 수신자가 사실상 없음.

### 2-3. 인프라 이슈

| 이슈 | 위험도 | 설명 |
|------|--------|------|
| CORS `["*"]` 설정 | 🔴 높음 | 모든 도메인에서 API 접근 가능. CSRF/데이터 탈취 위험 |
| daily-reminder 로그 없음 | 🟡 중간 | 크론잡 동작 여부 확인 불가. 알림 발송 실패 가능성 |
| DB Connection reset | 🟢 낮음 | 간헐적 발생, 배포 시 정상적인 연결 종료로 추정 |
| device_tokens 기본 언어 `zh` | 🟢 낮음 | 중국어 기본값 하드코딩 가능성, 다국어 지원 시 확인 필요 |

## 3. 개선 제안 및 우선순위 로드맵

### 🔴 Phase 1: 긴급 수정 (1주 내)

> 사용자 이탈을 직접 방지하는 버그 수정과 보안 이슈

#### 1-1. 체중 입력 키보드 이슈 수정
- **문제:** 숫자 키보드가 닫히지 않아 저장 버튼 접근 불가
- **해결:** `FocusScope.of(context).unfocus()` 또는 키보드 외부 탭 시 키보드 닫기 구현. 저장 버튼을 키보드 위에 표시하는 방법도 고려.
- **영향:** 체중 기록이 킬러 기능이므로 이 버그는 **사용자 이탈의 직접적 원인**

#### 1-2. 앱 성능 최적화 (렉/카드 이슈)
- **문제:** "有点卡", "一直没存进去" 등 성능 불만
- **점검 항목:**
  - 홈 화면 API 호출 최적화 (병렬 호출, 캐싱)
  - 불필요한 rebuild 방지 (setState 최적화)
  - 이미지 로딩 최적화
- **영향:** 첫 인상이 "느리다"면 재방문율 급감

#### 1-3. CORS 설정 보안 강화
- **문제:** `CORS_ORIGINS: ["*"]` — 모든 도메인에서 API 접근 가능
- **해결:** 실제 프론트엔드 도메인만 허용하거나, 모바일 앱 전용이므로 CORS를 더 엄격하게 설정
- **영향:** 보안 취약점 해소

#### 1-4. 회원가입 다국어 지원 확인
- **문제:** 회원가입 화면이 한국어만 표시되어 중국 사용자 이탈
- **해결:** 기존 다국어 시스템(i18n) 확인, 회원가입/로그인 플로우 중국어 번역 완성도 점검
- **영향:** 중국 시장이 메인 마케팅 채널(小红书)인 상황에서 치명적

### 🟡 Phase 2: 핵심 UX 개선 (2-3주)

> 사용률이 낮은 기능의 UX 개선 + 사용자 요청 반영

#### 2-1. 사료 기록 데이터 수정 기능 추가
- **문제:** 현재 삭제 후 재입력만 가능 → "每次都要删了重打有点点麻烦"
- **해결:** 사료/음수 기록 편집(edit) 기능 추가
- **DB 현황:** food_records 42건 중 수정 시도로 인한 중복 삭제/재생성 패턴 확인 필요
- **영향:** 사료 기록 사용률 16.8% → 개선 시 30%+ 기대

#### 2-2. 사료/음수 입력 UX 간소화
- **문제:** 사용률 각각 16.8%, 10.4%로 극히 낮음
- **분석:** 체중 기록(98.4%)과의 차이가 너무 큼 → 입력 허들이 높거나 동기부여 부족
- **해결 방안:**
  - 원탭 퀵 입력 (자주 사용하는 값 프리셋)
  - 홈 화면에서 바로 입력 가능한 위젯
  - 체중 기록 완료 후 "사료/음수도 기록하시겠어요?" 넛지
- **영향:** 핵심 건강 데이터 수집률 향상 → AI 분석 정확도 향상

#### 2-3. 푸시 알림 권한 요청 개선
- **문제:** 250명 가입 대비 device_tokens 2건 (0.8%)
- **심각성:** 1,991건의 알림이 발송되고 있지만, 실제 수신하는 사용자가 거의 없음
- **해결:**
  - 알림 권한 요청 타이밍 개선 (온보딩 완료 후, 또는 첫 체중 기록 후)
  - 알림의 가치를 먼저 설명하는 프리퍼미션 다이얼로그
  - "체중 기록 리마인더를 받으시겠어요?" 같은 구체적 제안
- **영향:** 리텐션 극대화의 핵심 채널 복구

#### 2-4. daily-reminder 서비스 로깅 및 모니터링
- **문제:** 로그가 전혀 없어 크론잡 정상 동작 여부 확인 불가
- **해결:**
  - 크론잡 실행 시 로그 출력 추가
  - 실행 결과(성공/실패/발송 건수) 기록
  - 실패 시 알림 시스템 (Slack webhook 등)
- **영향:** 알림 시스템 안정성 확보

### 🟢 Phase 3: 성장 기능 (1-2개월)

> 사용자 확대 및 engagement 강화

#### 3-1. Android 출시 (최우선)
- **근거:** 小红书 댓글의 **가장 많은 요청**(17건). 중국 시장 Android 점유율 80%+
- **방안:**
  - APK 직접 배포 (공식 웹사이트)
  - 중국 Android 마켓 등록: 华为应用市场, 小米应用商店, OPPO 软件商店
  - Google Play 등록 (글로벌 사용자)
- **기술:** Flutter 기반이므로 Android 빌드는 기술적으로 준비 완료
- **주의:** 중국 Android 마켓 별도 심사/계정 필요, 결제 시스템 차이

#### 3-2. AI 기능 발견가능성(Discoverability) 강화
- **문제:** AI 채팅 사용률 9.6% — 기능 존재를 모르거나, 접근 경로가 불명확
- **해결:**
  - 홈 화면에 AI 채팅 바로가기 카드 강화 (현재보다 눈에 띄게)
  - 체중 변화 감지 시 AI 자동 제안: "체중이 3일 연속 감소했어요. AI에게 물어볼까요?"
  - 첫 사용 유도 코치마크/툴팁
- **기대 효과:** AI 사용률 9.6% → 25%+

#### 3-3. 사료/음수 기록 Gamification
- **문제:** 일일 기록 5.6%, 스케줄 3.6% — 동기부여 부족
- **해결:**
  - 연속 기록 스트릭 (🔥 7일 연속 기록!)
  - 주간 기록 완성도 표시 (진행률 바)
  - BHI(건강지수) 점수에 기록 완성도 반영
- **영향:** 일일 활성 사용자(DAU) 증가, 데이터 밀도 향상

#### 3-4. 앱 내 FAQ / 도움말 추가
- **근거:** "要钱吗?", "AI准确吗?", "数据能保存几年?" — 반복되는 질문
- **해결:**
  - 설정 > 자주 묻는 질문(FAQ) 섹션
  - AI 면책조항 표시 (수의사 상담 권유)
  - 데이터 보관 정책 명시
- **영향:** 사용자 신뢰도 향상, CS 부담 감소

#### 3-5. WeChat 로그인 추가
- **근거:** 중국 시장 메인 타겟 시 WeChat은 필수 인증 수단
- **기술:** WeChat Open Platform SDK 연동 필요
- **주의:** WeChat 개발자 계정 심사 과정이 길 수 있음 (2-4주)
- **대안:** 이메일 로그인 경로를 더 쉽게 접근 가능하도록 우선 개선

### 🔵 Phase 4: 전략적 확장 (3개월+)

> 장기적 성장 전략

#### 4-1. 조류 종 확장 (문조, 잉꼬 등)
- **근거:** 小红书에서 문조 지원 요청 3건
- **방향:** 데이터 모델은 이미 species 필드 존재 → 종별 참고 데이터(정상 체중 범위, 식단 등)만 추가하면 확장 가능
- **시기:** 앵무새 기능 안정화 이후

#### 4-2. AI 업그레이드 (기존 설계 문서 참조)
- **참조:** `2026-02-28-ai-upgrade-design.md`
- SSE 스트리밍, GPT-4o, Vision 건강체크, RAG 강화, 티어 시스템
- AI 사용률이 현재 9.6%이므로, **AI 기능의 발견가능성 개선 (Phase 3-2)을 먼저 진행한 후** 업그레이드해야 ROI가 높음

#### 4-3. Firebase Analytics 구현 (기존 설계 문서 참조)
- **참조:** `2026-02-27-analytics-and-in-app-review-design.md`
- 이벤트 트래킹 + In-App Review 연동
- **필요성:** 현재 DB 직접 쿼리로만 사용률 파악 가능 → Analytics로 실시간 모니터링 필수

#### 4-4. 咕咕智宠 제휴 검토
- **근거:** 小红书에서 합작 제안 1건
- **방향:** 스마트 조류 용품 브랜드와의 제휴 → 하드웨어 연동 가능성 (자동 체중 측정 등)
